Major changes:

Between Stellaris version 2.0 and version 2.8, the primary hardware changes were not required physical ones, instead it changed how the software utilized that hardware, shifting from a single-threaded bottleneck to better multicore optimization to combat late-game lag.

While optimization helped, the primary way the game maintained playable speeds for older CPUs between updates 2.0 and 2.8 was through updates that reduced total pop growth in the galaxy, reducing the load on the CPU.

The transition to 64-bit allowed the game to handle significantly more data, improving performance for massive galaxies, particularly in the late game where 2.3 was 10-30% faster than previous versions.

The update 2.0 ("Cherryh") changed massive parts of the game that indirectly increased the hardware demands on the CPU and memory, update 2.0 solidified Stellaris as a CPU-heavy game that struggled with lag during massive late-game fleet battles, pathfinding, and population calculations.

Late-game performance in 2.8 was improved by optimizing "tick times" and implementing more efficient parallelization, meaning the game was better at spreading the load across multiple CPU cores rather than relying solely on one.


Why:

The transition from Stellaris 2.0 to 2.8.1 (and beyond) involved significant changes to hardware utilization primarily to address severe late-game performance issues caused by the "Le Guin" (2.2) update, which replaced the tile system with a complex, pop-heavy economic simulation.
